Hangman Challenge 

This challenge explores training a deep learning model to play Hangman by predicting the correct letter for masked words. Over the course of the challenge, I experimented with various architectures and training strategies—including LSTM, transformer, and double-Q learning—to determine the most effective approach. Ultimately, the transformer model delivered the best performance and was refined through increased epochs and parameter tuning.

The challenge consists of three main components, each documented in its respective notebook:
  • Transformer
  • Dataset Construction 
  • Tokenizer

Overview

I initially implemented an LSTM model, which showed promising results. Motivated by its potential, I switched to a transformer-based model that performed significantly better. I also experimented with double-Q learning; however, its performance was not as satisfactory. Returning to the transformer, I increased the number of training epochs and fine-tuned the model.

Although these efforts led to considerable improvements, I recognize that there is still ample room for optimization. With additional compute resources, further hyperparameter tuning (e.g., a grid search over learning rates, using a learning rate scheduler, modifying the BPE tokenizer merges, and refining dataset construction) could boost the win rate even further.

Why Transformer and Not LSTM?
  • While the initial experiments with LSTM showed potential, the transformer model outperformed LSTM by better capturing long-range dependencies and context within the input sequences.
  • The transformer architecture provided greater training efficiency and overall performance improvements, making it the preferred choice for this challenge.

Detailed Descriptions

1. Transformer 
   This document outlines the implementation of the transformer model used for the Hangman game.

   - Model Architecture:
       • Token Embedding: Converts input tokens into dense vector representations.
       • Positional Encoding: Adds sequence information to the embeddings.
       • Transformer Encoder: Processes the embedded inputs to capture complex contextual relationships.
       • Wrong Guesses challengeion: Integrates features representing wrong guesses.
       • Classifier: Combines the transformer output with the wrong guesses embedding to predict the masked letter.
   
   - Training Process:
       • The training loop includes loss computation, backpropagation, and checkpointing.
       • Increasing the number of epochs led to improved performance.

2. Dataset Construction 
   This document details the process of creating a diverse dataset to train the model. The approach was recently modified to improve the variety and richness of training examples.

   - Modified Masking Strategy:
       • For each word, examples are generated by masking all possible non-empty subsets of its distinct letters.
       • In each example, every occurrence of the selected subset of letters is replaced with an underscore ('_').
       • The masked letter is chosen as the most frequent letter (based on a predefined frequency order) among the masked letters.
   
   - Wrong Guesses Vector:
       • A one-hot encoded vector of size 26 is computed for each word, marking letters (from a predefined frequency order) that are not present in the word as wrong guesses—up to a maximum of 6.
   
   - Supervised Learning Approach:
       • I opted for a supervised learning strategy for dataset construction because it was the easiest and quickest way to achieve very high accuracy.
       • Alternative approaches, such as double-Q learning, struggled considerably, reinforcing the decision to use a supervised method.
   
   - Output Format:
       • The resulting dataset consists of tuples in the form (masked_word, masked_letter, original_word, wrong_guesses), which are saved as a CSV file for training.

3. Tokenizer
   This document describes the creation of a Byte Pair Encoding (BPE) tokenizer, which converts words into sequences of token IDs suitable for the transformer model.

   - BPE Tokenizer Algorithm:
       • The tokenizer iteratively merges the most frequent adjacent token pairs from the training corpus.
       • A vocabulary mapping tokens to unique IDs is generated after a predefined number of merge operations.
   
   - Rationale for BPE:
       • The idea behind using BPE was to help the model capture the context of consecutive letters in a word, thereby understanding word semantics more effectively.
       • In practice, I observed that using 100 merges in the BPE tokenizer resulted in significantly better performance compared to using only 50 merges, indicating that a richer vocabulary helps capture the nuances of the language better.
   
   - Tokenization Process:
       • A greedy, longest-match approach is used to convert words into sequences of token IDs.
       • This process is critical for preparing text data for the transformer model.

Experimentation and Future Work

Throughout the challenge, I experimented with various approaches:
  • LSTM Model: Provided a strong baseline.
  • Transformer Model: Achieved significant improvements in performance.
  • Double-Q Learning: Was explored but did not meet performance expectations.
  • Refined Transformer: Returning to the transformer model, I increased training epochs and fine-tuned the parameters.

While these experiments led to promising results, there is still significant potential for further optimization. Future work may include:
  • Hyperparameter Tuning:
       - Conducting a grid search over learning rates.
       - Implementing a learning rate scheduler.
       - Adjusting the maximum number of merges in the BPE tokenizer.
       - Refining the dataset construction process for even greater example diversity.
  • Transformer Architecture Adjustments:
       - Experimenting with the number of attention heads and layers.
       - Tweaking the classifier architecture, dropout ratios, and other parameters.

Each of these avenues could potentially significantly increase the win rate of the model, though comprehensive exploration was limited by available computational resources.
